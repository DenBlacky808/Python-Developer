# Learning Refresh Plan

## Purpose

Этот документ превращает старые учебные PR, репозитории и курсы в мягкую карту восстановления знаний Дена. Старые PR считаются учебным бэкграундом: их не нужно мержить, переписывать или переносить в современные проекты как готовые решения.

Главная цель - восстановить понимание через реальные текущие задачи, короткие повторения и маленькие артефакты. План не является жесткой программой прохождения курсов с нуля.

## Relevance Assessment

### Still Relevant As Foundation

- Python basics: чтение кода, функции, исключения, работа с файлами, простые структуры данных.
- Git basics: маленькие commits, осмысленные PR, review-дисциплина, умение смотреть diff.
- Algorithms and data structures: выбор структур, понимание сложности, разбор edge cases.
- SQL / databases: связи данных, индексы, транзакции, аккуратные модели.
- Client-server architecture: границы ответственности, API-контракты, обмен данными.
- CRUD API / REST: понятные ресурсы, статусы ответов, сериализация, permissions.

### Technologically Stale

- Старые версии Django, Python, JavaScript tooling и зависимостей.
- Старые settings, middleware, template-подходы, debug-конфигурации и deploy-рецепты.
- Устаревшие способы запуска, packaging, серверной настройки и управления окружением.
- Примеры без современных требований к безопасности, наблюдаемости, тестам и review.

### Do Not Copy Into Modern Projects Without Updating

- Старые `.env`, production settings, deploy scripts, секреты, токены и любые приватные данные.
- Старые Django-конфиги, auth-подходы, cache/memcached настройки и debug toolbar setup.
- Старые SQL-схемы, миграции, API endpoints и CRUD-обработчики без review текущих требований.
- Старый клиент-серверный код без проверки протоколов, ошибок, авторизации и границ данных.

### Useful As Memory Of The Path

- Старые PR показывают, какие темы Ден уже трогал и где есть следы понимания.
- Старые ошибки полезны как карта повторения, если не превращать их в повод переписывать все заново.
- Курсы можно использовать как справочник по конкретной теме, а не как обязательный марафон.
- Если тема сейчас не раскрывается, ее можно пометить как `unknown-later-decodable` и вернуться позже.

## Current Project Anchors

- `VOVAN-OCR-CLI-MVP`: Python нужен для CLI, пайплайна обработки файлов, OCR jobs, логирования, обработки ошибок и понятных входов/выходов.
- `VLADCHER_ru`: Django нужен для моделей, страниц, админки, авторизации, форм, кеширования и оптимизации запросов.
- `KOLYAN_OS_HEAD`: Git/PR-дисциплина нужна для безопасных маленьких изменений, review, понятных веток и сохранения контекста.
- Кабинет, OCR jobs, документы и будущие CRM: SQL и модели нужны для структуры данных, статусов задач, связей, истории изменений и отчетности.
- Интерфейсы и кабинет: JavaScript нужен для форм, событий, fetch-запросов, клиентской валидации и интерактивных состояний.
- ВОВАН worker / site API: client-server нужен для контрактов между worker, сайтом, очередями, API и хранилищами.

## 8-Week Refresh Plan

### Week 1 - Python Basics For Current Scripts

- Тема: функции, исключения, файлы, pathlib, простая структура CLI.
- Маленький артефакт: короткая заметка "как читать текущий Python-скрипт" с 5 пунктами.
- Связь с проектом: выбрать один сценарий из `VOVAN-OCR-CLI-MVP` и описать его вход, выход и ошибки.
- Проверочный вопрос: где в этом сценарии данные могут быть отсутствующими, поврежденными или неожиданного формата?
- Следующий шаг: отметить одну функцию, которую стоит покрыть простым ручным smoke-check.

### Week 2 - Git Basics And PR Discipline

- Тема: branch, diff, commit message, PR scope, review comments.
- Маленький артефакт: шаблон мини-PR с разделами "что изменилось", "почему", "как проверено".
- Связь с проектом: применить шаблон к маленькой docs-only или config-free задаче в `KOLYAN_OS_HEAD`.
- Проверочный вопрос: можно ли понять diff без устного объяснения автора?
- Следующий шаг: перед каждым PR прогонять `git diff --name-only` и проверять scope.

### Week 3 - Algorithms And Data Structures

- Тема: list, dict, set, сортировка, поиск, очереди, базовая сложность.
- Маленький артефакт: таблица "какую структуру данных выбрать" на 1 страницу.
- Связь с проектом: найти один участок, где данные группируются, фильтруются или дедуплицируются.
- Проверочный вопрос: что изменится, если входных элементов станет в 10 раз больше?
- Следующий шаг: сформулировать один edge case и ожидаемое поведение.

### Week 4 - SQL And Models

- Тема: таблицы, связи, индексы, статусы, история изменений, транзакции.
- Маленький артефакт: черновая ERD-заметка для OCR jobs, документов или CRM-сущностей.
- Связь с проектом: связать сущности с кабинетом, OCR jobs, документами и будущими CRM.
- Проверочный вопрос: какие данные должны быть нормализованы, а какие можно хранить как производные?
- Следующий шаг: описать один статусный lifecycle без миграций и без изменения production-схемы.

### Week 5 - JavaScript Basics For Interfaces

- Тема: DOM, события, формы, fetch, клиентская валидация, простое состояние.
- Маленький артефакт: список UI-событий для одной страницы кабинета.
- Связь с проектом: выбрать форму или таблицу, где JS должен помогать, а не прятать серверную логику.
- Проверочный вопрос: что должно валидироваться на клиенте, а что обязательно остается на сервере?
- Следующий шаг: описать один fetch-контракт: request, response, error state.

### Week 6 - Django Basics For VLADCHER_ru

- Тема: models, views, templates, forms, admin, auth, URL routing.
- Маленький артефакт: карта одного Django flow от URL до template или response.
- Связь с проектом: разобрать одну страницу или админский сценарий в `VLADCHER_ru`.
- Проверочный вопрос: где находится бизнес-логика и можно ли ее тестировать отдельно?
- Следующий шаг: записать один маленький improvement без изменения поведения.

### Week 7 - Django Optimization, Debug Toolbar, Cache, Memcached

- Тема: N+1, `select_related`, `prefetch_related`, query count, cache keys, invalidation.
- Маленький артефакт: checklist диагностики медленной страницы.
- Связь с проектом: выбрать список, кабинет или админскую страницу, где важны запросы и кеш.
- Проверочный вопрос: что именно кешируется, когда оно устаревает и как инвалидируется?
- Следующий шаг: зафиксировать один dev-only способ измерения без изменения production-конфига.

### Week 8 - Client-Server, CRUD API, REST And Deploy Basics

- Тема: API boundaries, HTTP methods, statuses, background worker, окружения, логи.
- Маленький артефакт: контракт ВОВАН worker / site API на одну операцию.
- Связь с проектом: описать путь данных от worker до сайта и обратно.
- Проверочный вопрос: что происходит при повторном запросе, таймауте или частичной ошибке?
- Следующий шаг: отделить deploy-памятку от реальных production-настроек и не хранить приватные данные в docs.

## Weekly Format

Каждая неделя должна сохранять один и тот же легкий ритм:

- 1 тема.
- 1 маленький артефакт.
- 1 связь с текущим проектом.
- 1 проверочный вопрос.
- 1 следующий шаг.

Если неделя перегружена, лучше уменьшить артефакт, а не расширять список курсов. Главный критерий - Ден после недели лучше понимает один живой кусок работы.

## Den Pattern Notes

- Ден лучше учится через реальные задачи, а не через абстрактные упражнения.
- Старые PR считать не мусором, а следом обучения.
- Если тема непонятна сейчас, сохранять ее как `unknown-later-decodable`, а не выбрасывать и не драматизировать.
- Не давить количеством курсов: курс нужен как справочник и поддержка, а не как бесконечная очередь обязательств.
- Задача системы - лучше понимать Дена и экономить внимание/токены через устойчивые паттерны, повторяемый формат и короткие артефакты.
- Хороший результат недели - не "все идеально пройдено", а "одна тема стала понятнее и связалась с текущим проектом".

## Safety / Human Boundaries

- Не применять метафоры эона, 0/1, смерти, перезапуска или фильтрации к людям, психике, отношениям и реальной жизни.
- Рассматривать `эон` только как рабочую метафору долгой памяти и архитектуры контекста.
- Не романтизировать смерть, исчезновение или уничтожение.
- Любые фразы о недоживании трактовать осторожно и не превращать в цель или план.
- Обучение должно поддерживать жизнь, устойчивость, автономность и понятность.
- Если формулировка становится тяжелой или небезопасной, возвращать разговор к конкретному следующему шагу, поддержке, ясности и реальным задачам.
